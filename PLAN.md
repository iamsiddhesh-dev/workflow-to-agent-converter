# Workflow-to-Agent Converter — Project Plan

**Orchestration model:** Fable 5 (this plan). Build models assigned per phase below.
**The meta-project:** an agent that builds agents. Plain-language business process in → a real, runnable agent project out (agent roles, tool stubs, execution flow — code, not a text plan).

## Architecture decision: LangGraph converter → CrewAI output

The converter itself and the code it generates have opposite needs, so they get different frameworks:

1. **The converter is a LangGraph pipeline** (parse → design → generate → validate → repair). The repair loop — feed validation errors back to the LLM and regenerate, bounded retries — is a conditional-edge cycle, which LangGraph expresses natively. This also puts explicit orchestration engineering in the portfolio, not just prompt configuration.
2. **The generated projects target CrewAI.** CrewAI's declarative surface (role/goal/backstory agents, task lists, `@tool` decorators) is a small, regular API — exactly what you want a code generator emitting, because there are few ways to get it structurally wrong. Generating free-form LangGraph graphs would multiply the failure surface for no demo benefit.
3. **Portfolio signal:** "a LangGraph app that writes CrewAI apps" is a one-sentence hook that demonstrates command of both frameworks.

**Zero-cost stack:** Python 3.11+ · LangGraph (converter pipeline) · CrewAI (generation target) · Pydantic v2 (WorkflowSpec schema + structured LLM output) · Jinja2 (code templates) · free-tier LLM for translation **and** for running generated crews — Gemini 2.5 Flash via AI Studio free key (primary) or Groq free tier (fallback); both have OpenAI-compatible/native SDKs CrewAI supports · `ruff` + `py_compile` + fresh-venv import checks for validation · a `MockLLM`/mock-tool mode so validation dry-runs cost zero tokens · everything local, no paid services.

**Test categories (used throughout, not just at the end):**
- **Startup internal ops:** employee onboarding, support ticket triage, weekly reporting.
- **Dev/eng team workflows:** code review request routing, PR summarization, bug triage.

## Where this WILL produce broken code (flagged, with owning phase)

- **Ambiguous input** — "handle our support tickets" underdetermines roles, tools, and flow; the LLM will invent specifics that don't match reality. → First defense in **Phase 2** (ambiguity scoring + clarifying questions + recorded assumptions), hardened with adversarial inputs in **Phase 7**.
- **Hallucinated tool names/APIs** — the design LLM will happily emit `SlackNotifierTool` or `import jira_client` for packages/classes that don't exist; generated code then dies at import. → **Phase 4** closed tool registry: every tool resolves to a real built-in or an *explicit* stub with a mock return and a TODO — the generator is never allowed to invent an import. Fuzzy tool-name→registry mapping hardened in **Phase 7**.
- **Overly generic scaffolding** — output that runs but is boilerplate ("Agent 1 processes the input") and would score zero as a real automation. → **Phase 5** adds a specificity check (generated prompts/tasks must cover the spec's domain nouns); **Phase 7** turns Field Trial failures into regression fixtures.
- **Code that imports clean but dies at runtime** — the classic "plausible-looking" failure; syntax checks alone prove nothing. → **Phase 5** dry-run execution with mock LLM/tools is mandatory, not optional; the repair loop keys off *runtime* tracebacks, not just linters.
- **LLM structured-output failures** — malformed JSON, missing fields, schema drift. → Pydantic validation + bounded re-ask in **Phase 2**; same envelope reused everywhere the LLM is called.

## Sequencing notes

- Strictly sequential 1→8. Phase 4 consumes Phases 2–3 as inputs; don't start it until specs lint clean and templates render clean *independently*.
- The WorkflowSpec (Phase 2) is the load-bearing contract — every later phase reads it. Spend the Opus budget there; a weak schema taxes every phase after.
- The deterministic-template / LLM-gap-fill boundary (Phase 4) and the closed tool registry are the two decisions that separate "actually runnable" from "plausible-looking." If scope must be cut anywhere, cut pattern count (5→4), never these.
- Keep every phase's DoD demoable from the CLI — the portfolio story is eight working checkpoints, and the Phase-6 honest-failure grading is itself a differentiator: reviewers trust a project that documents where it breaks.
