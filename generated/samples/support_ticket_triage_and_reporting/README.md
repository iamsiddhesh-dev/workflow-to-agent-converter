# Support Ticket Triage and Reporting

Generated by w2a from a `watcher` pattern.

Categorize incoming Zendesk support tickets, alert on-call staff for urgent issues, and generate a weekly analytical summary.

**Trigger:** A new ticket arrives in Zendesk.

## Pattern notes

main.py polls in a loop, calling crew.run_once() every --interval seconds; pass --once to run a single pass (used for validation).

## Agents
- **Ticket Classifier** (`triage_agent`) — Read incoming tickets, label them by category, and detect urgent issues.
- **On-Call Notifier** (`alert_agent`) — Trigger immediate alerts for high-priority issues via PagerDuty.
- **Operations Analyst** (`reporter_agent`) — Compile a weekly summary of ticket trends and volume.

## Tasks
- `classify_ticket` (agent: `triage_agent`)  Analyze the text of the incoming zendesk ticket to determine its category as bug, billing, or question. Evaluate the content for specific urgency keywords to output a label and a boolean status for the triage process.
- `generate_weekly_report` (agent: `reporter_agent`)  Gather all support ticket data recorded over the past seven days to analyze volume and category distribution. Produce a weekly analytical summary document that highlights identified trends and counts of bugs, billing requests, and general questions.
- `trigger_alert` (agent: `alert_agent`) — depends on classify_ticket  Receive the category label and urgency status from the triage process to decide if an action is required. If the status is urgent, execute a notification via the external system to alert the on-call staff and return a confirmation message.

## Tools
- **Zendesk Connector** (external) — Provides read access to incoming ticket content and allows for the updating of ticket labels after triage classification.. `MOCK_MODE` stub in `tools.py` (`# TODO: connect real Zendesk Connector`) — returns a canned payload until wired to the real system.
- **PagerDuty Notifier** (external) — Enables the transmission of urgent alerts to the on-call staff when critical issues are identified in the support queue.. `MOCK_MODE` stub in `tools.py` (`# TODO: connect real PagerDuty Notifier`) — returns a canned payload until wired to the real system.

## Assumptions carried from the spec
- The weekly report is generated on a set schedule at the end of the week.
- Urgent criteria includes keywords like 'down', 'broken', 'can't login', or 'losing money'.
