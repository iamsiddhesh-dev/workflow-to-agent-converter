# PR Summary Generator

Generated by w2a from a `sequential` pattern.

Generate a high-quality summary for pull requests to improve review quality, distinguishing between substantive changes and trivial fixes.

**Trigger:** A new pull request is opened or updated.

## Pattern notes

Tasks run strictly in dependency order, each feeding its output as context to the next.

## Agents
- **PR Analyst** (`diff_analyst`) — Determine the functional behavior change, identify risky areas, and verify test coverage in a PR.
- **Technical Writer** (`comment_generator`) — Synthesize PR findings into a clear, actionable comment for reviewers.

## Tasks
- `analyze_pr` (agent: `diff_analyst`)  Examine the pull request diff to classify it as trivial or substantive. Extract core behavior changes, identify potential risks such as migrations or auth config updates, and verify the presence of adequate test coverage to produce a technical assessment.
- `write_pr_comment` (agent: `comment_generator`) — depends on analyze_pr  Using the technical assessment from the diff analyst, draft a structured PR comment. For trivial fixes, provide a concise one-liner, and for substantive changes, construct a comprehensive summary that highlights functional impacts and risk factors for reviewers.

## Tools
- **Git Diff Reader** (external) — Retrieves the raw code changes from a pull request to enable the diff analyst to examine the scope of work.. `MOCK_MODE` stub in `tools.py` (`# TODO: connect real Git Diff Reader`) — returns a canned payload until wired to the real system.
- **PR Commenter** (external) — Publishes the generated summary or one-liner directly to the pull request thread to improve the review process.. `MOCK_MODE` stub in `tools.py` (`# TODO: connect real PR Commenter`) — returns a canned payload until wired to the real system.

## Assumptions carried from the spec
- The system has programmatic access to read PR diffs and write comments.
- The definition of 'trivial' (typo/comment) will be handled by the analyst agent's internal logic.

```
cp .env.example .env   # fill in GEMINI_API_KEY (or GROQ_API_KEY)
MOCK_MODE=1 python main.py path/to/input.txt   # dry run, no API calls, no key needed
python main.py path/to/input.txt               # real run
```

and every tool listed above as resolved to a builtin. What's stubbed: every
tool marked as a `MOCK_MODE` stub — each has a TODO and a canned mock payload,
