# CI Build Failure Analysis  [dev]
Analyze failed CI builds to distinguish between flaky tests and real regressions, notifying teams of regressions and reporting weekly statistics.
Trigger: A CI build fails.
Pattern: router

Agents:
  - analyzer (Failure Analyzer): Examine failed tests and determine if they are flaky or a real regression.
  - notifier (Incident Responder): Notify the appropriate team when a real regression is identified.
  - reporter (Metrics Analyst): Generate a weekly summary of failure trends.

Tasks:
  - analyze_failure @analyzer: Analyze which tests failed and compare them against historical data to classify them as 'flaky' or 'real regression'.
      -> A classification label for the failure (flaky or real) and the owning team identifier.
  - handle_regression @notifier <- analyze_failure: If analyze_failure indicates a real regression, ping the owning team responsible for the failing test file.
      -> Notification ping sent to the relevant team.
  - handle_flaky @notifier <- analyze_failure: If analyze_failure indicates a flaky test, label it as 'flaky, safe to retry'.
      -> Label applied to the CI build failure.
  - generate_weekly_digest @reporter: Compile a weekly report summarizing the ratio of flaky vs real build failures.
      -> A weekly statistical report on build failure trends.

Tools:
  - Test History Database [external]: Check if failing tests have a history of flakiness.
  - Team Notifier [external]: Ping the owning team when a real regression is found.

Assumptions:
  - The weekly report is generated on a separate, time-based schedule independent of individual build triggers.
  - The 'owning team' is identifiable via test file metadata.
  - The 'rough history' of flaky tests is accessible via the history database tool.
